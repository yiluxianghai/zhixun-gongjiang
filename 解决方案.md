# 工程监理巡视问题分类闭环管理智能体解决方案

## 一、项目概述

### 1.1 项目背景

工程监理现场巡视过程中，质量、安全、文明施工等问题频繁发现，但目前问题记录、分类定级、整改通知编写和闭环台账维护仍主要依赖人工完成。不同人员对问题描述口径、风险等级判断和整改要求表述存在差异，容易造成：

- **问题记录不标准**：不同监理人员描述同一问题时口径不一
- **责任划分不清晰**：问题描述模糊导致责任主体难以确定
- **整改要求不明确**：整改措施缺乏针对性，执行不到位
- **复查销项不及时**：问题闭环跟踪依赖人工，容易遗漏

尤其在大型项目、多专业交叉和问题数量较多的情况下，监理人员需要在现场照片、巡视记录、微信群、通知单和Excel台账之间反复整理，工作重复性强，问题数据也难以沉淀复用。

### 1.2 项目目标

构建"监理巡视闭环智能体"，围绕现场巡视问题的 **"发现—识别—分类—整改—复查—销项"** 全过程，开发一个轻量化AI辅助系统。实现：

1. 现场问题智能识别与标准化描述
2. 问题自动分类（类别+专业）与风险定级
3. 整改要求、复查要点、整改期限自动生成
4. 结构化问题卡片与闭环管理台账
5. 监理通知单、巡视记录、问题分析报告自动输出

### 1.3 设计原则

| 原则 | 说明 |
|------|------|
| 聚焦痛点 | 不追求大而全平台，聚焦"巡视问题闭环管理"核心场景 |
| 轻量落地 | 采用成熟技术栈，快速搭建可演示、可复制的原型 |
| 人机协同 | AI辅助生成+人工确认，确保专业准确性 |
| 数据沉淀 | 问题数据结构化存储，支持后续分析与复用 |
| 标准驱动 | 基于行业标准和管理规范构建知识库，确保输出规范性 |

## 二、需求分析

### 2.1 功能需求

#### 2.1.1 输入模块

| 输入方式 | 说明 |
|----------|------|
| 文字输入 | 监理人员手动输入问题描述 |
| 照片上传 | 上传现场巡视照片，通过OCR/多模态识别提取问题信息 |
| Excel导入 | 批量导入已有问题清单（支持标准模板） |
| 语音输入 | （远期）支持语音转文字输入 |

#### 2.1.2 智能分类模块

- **问题类别识别**：质量管理、安全管理、文明施工管理、进度管理等
- **专业类型识别**：土建、机电（水暖电）、装饰、消防、幕墙等
- **问题现象提取**：从非结构化描述中提取核心问题现象
- **标准化描述生成**：将口语化描述转为规范化的工程问题描述

#### 2.1.3 风险定级与整改建议模块

- **风险等级判定**：根据问题类型、严重程度、影响范围判定（一般/较大/重大）
- **整改要求生成**：基于规范和知识库生成针对性整改要求
- **整改期限建议**：根据风险等级和问题类型推荐整改时限
- **复查要点生成**：明确复查时需要检查的关键点
- **责任主体建议**：根据问题类型和专业自动建议责任单位

#### 2.1.4 闭环台账模块

- **问题编号自动生成**：按"项目代号-专业-年月-序号"规则编号
- **结构化字段管理**：问题编号、发现时间、问题部位、问题类别、风险等级、责任单位、整改要求、整改期限、复查状态、销项情况
- **状态流转管理**：待整改 → 整改中 → 待复查 → 已销项 / 未通过
- **状态统计看板**：未整改、整改中、待复查、已销项等状态汇总

#### 2.1.5 成果输出模块

| 输出类型 | 说明 |
|----------|------|
| 监理通知单 | 自动生成标准化监理通知单初稿 |
| 巡视记录 | 生成结构化巡视记录表 |
| 问题闭环台账 | 导出Excel格式闭环台账 |
| 问题分析报告 | 按类别/专业/风险等级统计分析 |

### 2.2 非功能需求

| 需求项 | 指标 |
|--------|------|
| 响应时间 | 单条问题AI分析 ≤ 10秒 |
| 并发能力 | 支持 ≥ 50个并发用户 |
| 可用性 | 系统可用率 ≥ 99.5% |
| 数据安全 | 支持本地化部署，数据不出项目 |
| 可扩展性 | 模块化设计，支持新增问题类型和规则 |

## 四、功能模块详细设计

### 4.1 输入模块

#### 4.1.1 文字输入

监理人员通过前端界面输入问题描述，支持以下辅助字段：

```json
{
  "project_name": "XX项目",
  "inspection_area": "A区地下室",
  "inspector": "王永春",
  "inspection_date": "2025-07-14",
  "raw_description": "地下室顶板模板支撑体系，扫地杆缺失，部分立杆间距偏大",
  "photo_urls": []
}
```

#### 4.1.2 照片上传（OCR+多模态）

- 前端支持多张照片上传，支持拖拽和粘贴
- 后端调用OCR服务提取照片中的文字信息
- 将OCR文本+原图描述交由LLM进行多模态分析
- 提取问题部位、问题现象、可见风险等信息

#### 4.1.3 Excel批量导入

支持标准Excel模板导入，模板字段包括：序号、发现日期、巡视区域、问题描述、发现人等。系统逐条调用AI分析模块进行处理。

### 4.2 智能分类模块

#### 4.2.1 问题分类体系

**一级分类（问题类别）**：

| 代码 | 类别 | 典型问题 |
|------|------|----------|
| QM | 质量管理 | 钢筋绑扎不规范、混凝土浇筑缺陷、防水施工不达标 |
| SM | 安全管理 | 临边防护缺失、脚手架搭设不规范、临时用电违规 |
| CM | 文明施工管理 | 现场材料堆放混乱、扬尘控制不到位、噪音超标 |
| PM | 进度管理 | 关键工序滞后、施工人员不足 |

**二级分类（专业类型）**：

| 代码 | 专业 | 典型关联 |
|------|------|----------|
| CIV | 土建 | 钢筋、模板、混凝土、砌体、防水 |
| MEP | 机电（水暖电） | 管道安装、电气布线、设备安装 |
| DEC | 装饰 | 抹灰、涂饰、吊顶、地面 |
| FIR | 消防 | 消防管道、喷淋系统、防火门 |
| CUR | 幕墙 | 龙骨安装、面板固定、密封胶 |

#### 4.2.2 分类实现策略

采用 **"知识库规则匹配 + LLM语义理解"** 混合策略：

1. **规则匹配优先**：从知识库中匹配关键词和规则，快速分类
2. **LLM兜底**：规则无法匹配时，调用LLM进行语义分析
3. **置信度评估**：当LLM置信度低于阈值时，标记为"需人工确认"

### 4.3 风险定级与整改建议模块

#### 4.3.1 风险等级标准

| 等级 | 判定标准 | 整改时限 | 处置要求 |
|------|----------|----------|----------|
| 一般 | 局部轻微缺陷，不影响结构安全和使用功能 | 7天内 | 下发整改通知，限期整改 |
| 较大 | 存在质量安全隐患，可能造成质量事故或安全事故 | 3天内 | 立即通知整改，暂停相关工序 |
| 重大 | 严重安全隐患或已发生质量事故 | 立即 | 立即停工，报告建设单位和主管部门 |

#### 4.3.2 整改建议生成

基于知识库中的**整改模板库**，结合LLM生成针对性整改要求：

```
输入：问题类别=安全管理, 专业=土建, 问题描述=模板支撑体系扫地杆缺失
知识库匹配模板：模板支撑体系整改模板
LLM生成：
  整改要求：1.立即补充安装扫地杆，确保距地面≤200mm；
           2.检查所有立杆间距，超过规范要求的进行调整；
           3.全面检查剪刀撑设置情况，缺失部位补齐。
  整改期限：3天内完成
  复查要点：1.扫地杆安装位置和数量；2.立杆间距是否符合方案；
           3.剪刀撑设置是否完整；4.整体稳定性检查。
  责任主体：总承包单位（模板工程班组）
```

### 4.4 闭环台账模块

#### 4.4.1 问题状态流转

```
┌─────────┐     整改通知      ┌─────────┐    整改完成     ┌─────────┐
│ 待整改  │ ───────────────→ │ 整改中  │ ────────────→ │ 待复查  │
└─────────┘                   └─────────┘                └────┬────┘
     │                             │                           │
                     ┌────────────┼───────────┐               │
                     │超期预警     │整改反馈    │        ┌──────┴──────┐
                     ▼            ▼           │        │             │
               ┌──────────┐  ┌────────┐      │   复查通过    复查不通过
               │ 超期未整改 │  │补充资料 │      │        │             │
               └──────────┘  └────────┘      │   ┌─────────┐  ┌─────────┐
                                              │   │ 已销项  │  │退回整改 │
                                              │   └─────────┘  └─────────┘
```

#### 4.4.2 问题编号规则

```
格式：[项目代号]-[专业代码]-[年月]-[序号]
示例：XXPT-CIV-2507-0001（XX项目-土建-2025年7月-第1条）
```

### 4.5 成果输出模块

#### 4.5.1 监理通知单

自动生成标准化监理通知单，包含：通知单编号、工程名称、致（施工单位）、事由、问题描述、整改要求、整改期限、监理单位（章）、日期等要素。

#### 4.5.2 巡视记录

生成结构化巡视记录表，包含：巡视日期、巡视区域、巡视人员、发现问题汇总、问题详情、处理意见等。

#### 4.5.3 问题分析报告

按类别、专业、风险等级、责任单位等维度进行统计分析，生成趋势图和高频问题清单。

## 十四、风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| LLM分析准确率不足 | 问题分类/定级不准确 | 知识库规则+人工确认双重保障 |

| 数据安全顾虑 | 项目数据泄露 | 支持私有化部署，数据不出项目 |
